# Developer Experience Audit

Date: 2026-03-01

## API Surface Issues

### Must fix — leaking internals

1. **Mermaid/code internals in public barrel** — `sanitizeMermaidDefinition`, `buildMermaidConfig`, `buildClassDefs`, `injectClassDefs`, `buildCodeTheme` are all exported from `tycoslide-components`. These are implementation details. Remove from `index.ts`.

2. **Theme-default leaks helpers via `export *`** — `headerBlock`, `contentBody`, `masteredSlide` leak through `export * from './layouts.js'`. Replace with explicit named exports.

3. **`LINE_SHAPE`** — pptxgenjs implementation detail, exported from core via wildcard. Remove.

4. **`extractInlineText`** — Dead code in `syntax.ts`, never imported anywhere. Remove.

### Design improvements

5. **~~`schema.text()` and `schema.image()` wrappers~~** — REJECTED after deep analysis. `component.schema` is the single mechanism. Reasons:
   - `schema.card()` is architecturally impossible without duplication or circular deps (schema module is at bottom of dependency chain, can't import from components package).
   - Components themselves use `textComponent.schema` internally (card, quote, testimonial reference it for their own param definitions) — it's the lingua franca for component-to-component type sharing.
   - Layout files already import component definitions and DSL functions from the same package in the same import statement — zero extra coupling.
   - `.schema` is auto-generated by `defineComponent` from the component's own body/params — one source of truth.
   - Semantic signal: `title: textComponent.schema` = "feeds a text component." `variant: schema.string()` = "config string." The distinction is real and useful.
   - **Rename to `.type` considered and rejected**: `.type` is the discriminant field on every node in the system (`TextNode.type`, `ComponentNode.type`, etc.). `textComponent.type` would read as "what kind of node is this?" — wrong meaning. `.schema` is unambiguous (nothing else in the codebase has a `.schema` field). If Zod is ever replaced, `.valueType` would be the natural rename.
   - **Convention to document**: Use `xComponent.schema` for params that feed components. Use `schema.*` helpers for non-component config values (strings, enums, numbers).

6. **~~DSL function signature patterns~~** — NOT NEEDED. Four patterns exist (body-first, props-only, optional-props, container-overload). The split maps to genuine domain differences (text has a body, card has named props, containers have children). Container overload is structurally safe — discriminated by `type`/`componentName` field presence. TypeScript catches misuse at compile time. One minor polish: `grid(number, ...children)` "magic number first" overload could be deprecated in favor of `grid({columns: 3}, ...children)`.

7. **`ComponentNode` exported twice from core** — Via wildcard from `nodes.ts` and explicitly from `registry.ts`. Minor cleanup.

8. **Wildcard exports dump ~60 symbols from `types.ts`/`nodes.ts`** — A curated explicit export list would be cleaner long-term.

### Naming

9. **`Component` is PascalCase** while every other const-object "enum" is `UPPER_SNAKE_CASE`. Reads better as `Component.Card` than `COMPONENT.CARD` — arguably correct, but inconsistent.

## Documentation Issues

### Critical gaps

1. **No local theme development workflow** — `themes.md` only covers npm-published themes. No `npm link`, `file:` dependency, or workspace setup documented. A theme developer hits a dead end after creating files.

2. **Multi-slot syntax (`::right::`) missing from built-in layout docs** — `two-column`, `comparison`, `bio` show "Slots: left, right" but never show the markdown syntax to fill them. Only appears 200 lines deep in the custom layout section.

3. **YAML complex types never shown** — `agenda` needs `items: string[]`, `cards` needs `cards: object[]`, `team` needs `members: object[]`. No YAML array/object examples anywhere.

### Significant gaps

4. **`themes.md` step-by-step creates `index.ts` referencing `components.ts` and `layouts.ts`** that are never created in the guide.

5. **No Node.js prerequisite** — Never mentioned. Markdown authors (target non-technical) will not know they need it.

6. **`register()` overloading unclear** — Docs show both `register(single)` and `register(array)` in different places without stating both work.

7. **Image path resolution undocumented** — Relative to markdown file? CWD? Absolute? Never stated.

8. **Canvas API underspecified** — Viewport size, available fonts, temp file lifecycle not documented.

## `--render-scale` Verdict

Keep it numeric, do not make it an enum.

- Playwright `deviceScaleFactor` accepts any positive number, and values 1–4 are all genuinely useful.
- An enum (`low|medium|high`) would be over-constraining without adding clarity.
- The help text `(1=fast, 2=retina, 3=print)` is misleading — implies only 3 values and uses imprecise labels.
- **Fix**: Change help to `pixel density multiplier for code blocks and diagrams (default: 2)`. Optionally clamp at max 4 to prevent OOM.
