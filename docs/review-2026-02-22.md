# Architectural Review — 2026-02-22

Three parallel architect reviews of `packages/core/`: structure, component system, and tests.

---

## Agreed Directory Structure

4 top-level dirs. Inside `core/`, all files in sub-dirs (no loose files alongside dirs).

```
src/
  cli/                         # entry point
    index.ts
    build.ts
    themeLoader.ts

  components/                  # renamed from dsl/
    index.ts
    primitives.ts              # image, line, shape, slideNumber
    containers.ts              # row, column, stack, grid
    text.ts                    # label, text, prose
    card.ts
    quote.ts
    table.ts
    mermaid.ts
    document.ts

  utils/                       # shared pure helpers
    assets.ts
    image.ts
    log.ts
    parser.ts
    font.ts                    # renamed from text.ts (eliminates name collision)
    units.ts

  core/                        # engine internals, all in sub-dirs
    model/                     # leaf types — no outward deps
      types.ts
      nodes.ts
      bounds.ts
      syntax.ts                # renamed from mdast.ts
      schema.ts

    rendering/                 # output pipeline
      registry.ts
      presentation.ts
      pptxConfigBuilder.ts
      pptxRenderer.ts

    layout/                    # browser-based CSS flexbox measurement
      layoutHtml.tsx
      measurement.ts
      pipeline.ts
      validator.ts

    markdown/                  # markdown → presentation compilation
      slideParser.ts
      slotCompiler.ts
      documentCompiler.ts

  index.ts                     # barrel export
```

**Dependency flow:** `model/` is leaf → `markdown/` and `layout/` depend on model → `rendering/` depends on everything.

## Detailed Findings

### Structure

**`core/` is overloaded** — mixes foundation types (types.ts, nodes.ts, bounds.ts), registry system (registry.ts), and PPTX output (pptxRenderer.ts, pptxConfigBuilder.ts). PPTX files depend on utils/, so core/ is not a leaf.

**Dependency direction after proposed restructure:**
```
cli/ --> markdown/ --> registry/ --> core/
              |            |           ^
              v            v           |
         components/ ------+-----------+
              |                        |
              v                        |
         layout/ --> utils/ -----------+
              |
              v
         render/ --> utils/ --> core/
```

**`dsl/document.ts` imports from `markdown/slotCompiler.ts`** — cross-layer dependency. Narrow (one function), tolerable. Don't duplicate.

**`text.ts` name collision** — `dsl/text.ts` (component) and `utils/text.ts` (font helpers) share same filename.

### Component System

**Overall: remarkably consistent.** Component enum usage is perfect. Token completeness verified for all 7 components. Semantic zeros pattern followed. No cross-component violations.

**`as any` inventory:**

| Location | Risk | Notes |
|----------|------|-------|
| `table.ts:120` | HIGH | Entire expand function cast |
| `registry.ts:304` | LOW | Props are `unknown` by design at generic boundary |
| `registry.ts:325,328` | LOW | Token bridging at generic boundary |
| `primitives.ts:78,128` | MEDIUM | Line/Shape cast props instead of using typed overload |

**Card gap tokens:** Card has `gap` AND `textGap`. Only `textGap` is used in expansion. `gap` is dead configuration. Quote uses `gap` correctly (single gap token). Inconsistent between the two.

### Tests

**Keep flat `test/` directory** — 21 files at this scale is navigable. The real problem is naming, not location.

**Three assert import patterns, two test function styles (`test()` vs `it()`)** — cosmetic but adds friction.

**Zero-coverage files:** pptxRenderer.ts (316 lines), measurement.ts (190 lines), pipeline.ts (164 lines), presentation.ts (278 lines).

**`layoutHtml.test.ts` is the outlier** — imports from dist/, creates own mock theme, highest risk of drift.
